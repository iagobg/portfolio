## Visão geral

Este portfólio também é um projeto: um site em Next.js orientado por conteúdo que apresenta trabalhos de software por meio de uma interface original inspirada em editores. O shell usa ideias úteis de workspace, como explorador, abas, breadcrumbs, paleta de comandos e barra de status, enquanto as páginas continuam semânticas e legíveis.

<Decision title="Portfólio como sistema de conteúdo">Adicionar um projeto deve significar adicionar um arquivo MDX e screenshots, não editar arrays de navegação ou componentes de página.</Decision>

## O problema

O objetivo era criar um portfólio memorável para desenvolvedores sem esconder os estudos de caso. Um clone literal de editor seria confuso, e um conjunto estático de cards ficaria chato de manter conforme os projetos crescessem.

## Arquitetura

Projetos vivem como arquivos MDX com frontmatter estruturada. Os mesmos metadados alimentam explorador, arquivo, busca, filtros, paleta de comandos, páginas de projeto, sitemap e navegação anterior/próximo.

| Área | Decisão |
| --- | --- |
| Framework | Next.js App Router |
| Conteúdo | Arquivos MDX locais com frontmatter |
| Estilo | Tailwind CSS e paleta customizada inspirada em editor |
| Localização | Rotas em inglês sob `/en` e rotas em português sob `/pt-BR`, com `/` redirecionando para `/pt-BR` |
| Fonte de dados | Conteúdo no filesystem, sem banco |

## Decisões técnicas

A implementação mantém Server Components como padrão e usa Client Components apenas para interações como paleta de comandos, drawer mobile, filtros e troca de idioma. A busca usa metadados leves em vez de carregar todo o corpo MDX no navegador.

<Callout title="Padrão de extensão">Esta entrada serve para copiar ao adicionar projetos futuros: primeiro frontmatter, depois um estudo de caso legível, depois screenshots.</Callout>

## Camada modular de deploy

O portfólio acabou virando a porta de entrada pública para vários projetos independentes, então o modelo de deploy precisava evitar que toda atualização virasse um restart do stack inteiro. A forma final separa os repositórios públicos das aplicações de um plano de controle privado de deploy:

| Camada | Responsabilidade |
| --- | --- |
| Repositórios dos projetos | Código da aplicação, Dockerfiles e workflows do GitHub Actions |
| GitHub Container Registry | Armazena uma imagem por projeto |
| Diretório privado de deploy | Compose overlays, rotas do Caddy, env files de runtime e volumes |
| VPS na Oracle | Roda Caddy e containers dos projetos em uma rede Docker interna |

Cada projeto é representado por dois pequenos arquivos privados de deploy: um overlay do Compose para o serviço e uma rota do Caddy para o subdomínio. O portfólio fica no domínio raiz, enquanto os projetos usam subdomínios como `futemon.<domínio>` e `livetrivia.<domínio>`.

<Decision title="Um projeto, um redeploy">Um push no Futemon deve publicar e recriar apenas o container do Futemon. O portfólio e os outros projetos devem continuar rodando.</Decision>

Os workflows do GitHub Actions publicam imagens no GHCR e depois acessam a VPS por SSH para rodar uma atualização direcionada com Compose:

```text
docker compose -f compose.yml -f projects/<service>.compose.yml pull <service>
docker compose -f compose.yml -f projects/<service>.compose.yml up -d --no-deps <service>
```

`--no-deps` evita recriar serviços não relacionados. Os segredos de runtime ficam na VPS em env files privados, então os repositórios públicos continuam reutilizáveis sem carregar credenciais ou estado específico do ambiente.

## Aprendizados de produção

O trabalho de deploy expôs problemas que raramente aparecem em um build local: atraso em delegação de DNS, validação de certificados pelo Caddy, permissões do socket Docker em sessões SSH do GitHub Actions, acesso privado ao GHCR e compatibilidade de imagens ARM64 nas instâncias Ampere da Oracle.

Esses problemas moldaram o setup final. Os workflows agora geram imagens ARM64 em runners ARM64 hospedados pelo GitHub, o Caddy roda em container junto com o stack, e a documentação de deploy separa setup inicial da VPS de redeploys por projeto.

<Limitation title="Limite operacional">O plano de controle de deploy é intencionalmente privado. O portfólio documenta a arquitetura, mas segredos, env files reais e estado específico da VPS vivem fora do repositório público.</Limitation>

## Estado atual

O site suporta rotas em inglês e português brasileiro, seleção automática no primeiro acesso por preferência de idioma do navegador, estudos de caso em MDX, metadados de SEO, navegação responsiva, publicação de imagens Docker e deploy modular na VPS para projetos irmãos.

## Aprendizados

A metáfora de editor funciona melhor quando melhora a navegação em vez de virar decoração. O shell pode ser técnico e familiar enquanto o conteúdo principal continua sendo web normal e acessível.

A camada de deploy seguiu o mesmo princípio: usar ferramentas familiares de um jeito que reduza coordenação. Um novo projeto deve precisar de um Dockerfile, um overlay do Compose, uma rota do Caddy e um workflow, não de um redesenho do portfólio inteiro.
